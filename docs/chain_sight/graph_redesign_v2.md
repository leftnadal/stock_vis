## 진단 요약 (현황의 핵심 문제)

현재 구현에서 발견된 구조적 결함은 다음과 같다.

| 문제 | 현재 상태 | 영향 |
|------|-----------|------|
| 캔버스 고정 400px | MarketGraphCanvas height 하드코딩 | 노드가 좁게 몰려 겹침 |
| 라벨 5자 절단 | `node.id.slice(0, 5)` | GOOGL → GOOGL 괜찮지만 긴 이름 티커 손실 |
| 엣지 동시 등가 표시 | SUPPLIES_TO부터 PRICE_CORRELATED까지 동일 시각 무게 | 공급망과 뉴스 상관의 구분 불가 |
| center 노드 크기 차이 미세 | center 16px vs 이웃 6~14px — 최대 차이 10px | 중심이 한눈에 안 들어옴 |
| 위치 의미 없음 | d3 force-directed = 충돌 최소화 알고리즘 | "왼쪽 = 공급망, 오른쪽 = 경쟁사" 같은 공간 개념 없음 |
| 첫인상 카피 부재 | "섹터를 선택하세요" — 정보 없음 | 사용자가 무엇을 기대해야 할지 모름 |
| 관계 토글 없음 | MarketGraphCanvas에 필터/토글 없음 | 5종 엣지를 모두 끄거나 켤 수 없음 |
| 빈 상태 최소화 | 텍스트 1줄만 | 다음 행동 유도가 없음 |

## 1. 시멘틱 방사형 레이아웃 (Semantic Radial Layout) 명세

### 1-1. 설계 원칙

**Why**: force-directed 레이아웃은 "충돌 최소화" 물리 알고리즘이므로 노드 위치에 의미가 없다.
사용자가 "어디를 보면 공급망인지, 어디를 보면 경쟁사인지"를 공간으로 파악할 수 있어야
멘탈 모델이 형성된다. 시멘틱 방사형은 관계 타입을 각도(시계 방향)로 고정 매핑하여
반복 사용만으로 직관이 쌓이게 한다.

### 1-2. 각도-관계 매핑 (6방향 배치)

```
                        0° / 360°
                    (12시 = 공급망 상류)
                    SUPPLIES_TO 방향

              공급망 상류                 공급망 상류
              (SUPPLIES_TO)              (CUSTOMER_OF)
      330°  ──────────────────────────  30°
           /          ↑                  \
          /       ┌───────┐               \
  270°  ━━       │       │        ━━━  90°
(경쟁사)  \     │  CENTER │      /  (Peer/테마)
  COMPETES_WITH  │ 노드   │    PEER_OF / HAS_THEME
          \       └───────┘       /
           \          ↓          /
      210°  ──────────────────────  150°
             (동시출현/가격상관)
              CO_MENTIONED /
              PRICE_CORRELATED
                   180°
               (6시 = 약한 신호)
```

### 1-3. 각도 구간 상세 명세

| 각도 범위 | 관계 타입 | 배치 의미 | 시각 처리 |
|-----------|-----------|-----------|-----------|
| 315° ~ 45° (12시 ± 45°) | SUPPLIES_TO | center에게 공급하는 상류 기업 | 실선 오렌지, 화살표 center 방향 |
| 45° ~ 135° (3시 ± 45°) | CUSTOMER_OF, PEER_OF | 고객(하류) + 동종 비교 | CUSTOMER_OF 오렌지, PEER_OF 파랑 |
| 135° ~ 225° (6시 ± 45°) | CO_MENTIONED, PRICE_CORRELATED | 약한 시장 신호 관계 | 점선 보라/회색, 낮은 불투명도 |
| 225° ~ 315° (9시 ± 45°) | COMPETES_WITH, HAS_THEME | 직접 경쟁 + 테마 공유 | 실선 빨강(경쟁), 점선 틸(테마) |

### 1-4. 반경(Ring) 위계

```
Ring 구조 (center 기준 동심원):

                         Ring 2 (외곽)
                     ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌
               Ring 1 (1차 이웃)
           ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
        ┌──────────────────────────────┐
        │        CENTER NODE           │  r = 28px
        │         (핵심 종목)          │  테두리 2.5px 흰색
        └──────────────────────────────┘
           반경 160px → 1차 이웃 배치   r = 14~20px (pagerank 비례)
                반경 280px → 2차 이웃   r = 8~12px  (pagerank 비례)

Ring 1 (반경 160px): truth_score 상위 6~8개, 투명도 1.0
Ring 2 (반경 280px): 나머지 이웃 + 2-depth, 투명도 0.7
```

### 1-5. 노드 크기 위계 (3단계 고대비)

| 단계 | 대상 | 반지름 | 불투명도 | 외곽선 |
|------|------|--------|---------|--------|
| Level 0 | center | 28px | 1.0 | 흰색 3px |
| Level 1 | 1차 이웃 (Ring 1) | 14~20px (pagerank) | 1.0 | 흰색 1.5px |
| Level 2 | 2차 이웃 (Ring 2) | 8~12px | 0.7 | 없음 |

**Why**: 현재 center 16px vs 이웃 6~14px 차이가 너무 작다.
Level 0을 28px로 올리면 center가 2배 이상 커져 즉각 인지된다.

### 1-6. 좌표 계산 방식 권고

**권고: 수동 좌표 계산 (Preset Position) 방식 채택**

Why: react-force-graph-2d의 d3 force는 `fx`, `fy` 고정 좌표를 지원한다.
관계 타입이 확정된 순간 각도와 반경으로 `(cx + R*cos(θ), cy + R*sin(θ))`를
계산하여 `fx`/`fy`에 주입하면 시멘틱 위치가 고정된다.
d3 force는 `fx`/`fy`가 있는 노드를 위치 고정으로 처리하므로
"공급망은 12시" 원칙이 항상 유지된다.

d3-force를 완전히 끄면 안 된다. center 주변에 여러 관계 타입 노드가 몰릴 때
같은 각도 구간 내 Y축 분산은 d3 charge force가 담당한다.
따라서 각도는 수동 고정, 동일 구간 내 반경·Y 분산은 d3 charge에게 위임한다.

```
의사코드 (JavaScript):

function calcPosition(relType, index, total, cx, cy) {
  const ANGLE_MAP = {
    SUPPLIES_TO:       { baseAngle: -90, spread: 90  },  // 12시 ± 45
    CUSTOMER_OF:       { baseAngle:   0, spread: 90  },  // 3시 ± 45
    PEER_OF:           { baseAngle:   0, spread: 90  },  // 3시 동일 구간
    COMPETES_WITH:     { baseAngle: 180, spread: 90  },  // 9시 ± 45
    HAS_THEME:         { baseAngle: 180, spread: 90  },  // 9시 동일 구간
    CO_MENTIONED:      { baseAngle:  90, spread: 90  },  // 6시 ± 45
    PRICE_CORRELATED:  { baseAngle:  90, spread: 90  },  // 6시 동일 구간
  };

  const cfg   = ANGLE_MAP[relType] || { baseAngle: 90, spread: 90 };
  const step  = total > 1 ? cfg.spread / (total - 1) : 0;
  const angle = (cfg.baseAngle - cfg.spread / 2 + step * index) * (Math.PI / 180);
  const R     = 160;  // Ring 1 반경

  return {
    fx: cx + R * Math.cos(angle),
    fy: cy + R * Math.sin(angle),
  };
}
```

## 2. 관계 토글 칩 바 명세

### 2-1. 레이아웃 위치

그래프 캔버스 상단 내부 (캔버스 영역 좌상단 오버레이) 또는
캔버스 바로 위 섹터 바와 그래프 사이에 독립 행으로 배치한다.
캔버스 내부 오버레이 방식은 터치 타겟 충돌이 있으므로
**캔버스 바로 위 독립 행 배치**를 권장한다.

```
┌──────────────────────────────────────────────────────────────────┐
│ 섹터 바 (SectorBar — 기존 유지)                                  │
├──────────────────────────────────────────────────────────────────┤
│ ┌────────────┐ ┌────────────┐ ┌──────────┐ ┌──────────┐ ┌─────┐ │
│ │ 공급망  ON │ │ 경쟁  ON   │ │Peer  ON  │ │뉴스  ON  │ │테마 │ │
│ │ ━━━━━ #F97 │ │━━━━━ #EF4 │ │─── #3B8  │ │╌╌╌ #A85  │ │OFF  │ │
│ └────────────┘ └────────────┘ └──────────┘ └──────────┘ └─────┘ │
│                                          [전체 켜기] [전체 끄기] │
├──────────────────────────────────────────────────────────────────┤
│ 그래프 캔버스                                                     │
└──────────────────────────────────────────────────────────────────┘
```

### 2-2. 칩 목록 — 레이블, 기본 상태, 색상 매핑

| 칩 레이블 | 대응 관계 타입 | 기본 상태 | 칩 색상 (ON) | 선 스타일 힌트 |
|-----------|---------------|-----------|-------------|---------------|
| 공급망 | SUPPLIES_TO, CUSTOMER_OF | ON | #F97316 (오렌지) | 실선 굵음 |
| 경쟁 | COMPETES_WITH | ON | #EF4444 (빨강) | 실선 보통 |
| Peer | PEER_OF | ON | #3B82F6 (파랑) | 실선 보통 |
| 뉴스 | CO_MENTIONED | ON | #A855F7 (보라) | 점선 |
| 가격상관 | PRICE_CORRELATED | OFF | #9CA3AF (회색) | 점선 |
| 테마 | HAS_THEME | OFF | #14B8A6 (틸) | 긴 점선 |

**Why 기본 OFF (가격상관, 테마)**:
가격상관과 테마는 엣지 수가 많고 신호 강도가 낮다.
첫 진입 시 가장 의미 있는 구조적 관계(공급망·경쟁·Peer)만 보여주는 것이
복잡도를 줄이는 가장 빠른 방법이다.

### 2-3. 칩 시각 명세

```
ON 상태 칩:
┌────────────────────┐
│ ━━  공급망    × │   ← 좌측: 엣지 색상 라인 샘플 8px, 우측: 끄기 아이콘
│                    │   배경: 해당 색상 10% 불투명도
│                    │   테두리: 해당 색상 70%
└────────────────────┘

OFF 상태 칩:
┌────────────────────┐
│     가격상관   + │   ← 색상 라인 없음, 텍스트 dim, 우측: 켜기 아이콘
│                    │   배경: #F3F4F6 (라이트) / #374151 (다크)
│                    │   테두리: #D1D5DB (라이트) / #4B5563 (다크)
└────────────────────┘

크기: height 32px, padding 8px 12px, border-radius 16px (pill)
폰트: 12px medium
간격: gap 8px
```

### 2-4. 인터랙션 규칙

**다중 선택 방식 (멀티 토글)**

Why: 단일 선택(라디오) 방식은 "공급망 + 경쟁 동시 보기" 유즈케이스를 막는다.
투자자는 "TSMC가 공급망으로도 연결되고 경쟁 뉴스에도 나오는지"를 동시에 보고 싶다.
따라서 각 칩은 독립 ON/OFF 토글이다.

**모두 끄면 어떻게 되는가**: 전체 끄기 상태에서는 노드만 표시하고 엣지를 모두 숨긴다.
"연결선 없이 종목 위치만 보기" 상태가 되며, 상단에
"관계 필터가 모두 꺼져 있습니다. 칩을 켜면 연결선이 나타납니다" 안내 토스트를 1회 표시한다.

**전체 켜기 / 전체 끄기 버튼**: 칩 바 우측 끝에 텍스트 버튼으로 배치.
"전체 켜기"는 모든 칩을 ON, "전체 끄기"는 모든 칩을 OFF.

### 2-5. 모바일 처리

```
모바일 (<768px):

┌─────────────────────────────────────────────────────┐
│ ← 스크롤 →                                           │
│ [공급망 ON] [경쟁 ON] [Peer ON] [뉴스 ON] [테마 ...]│
│  overflow-x: auto, scrollbar 없음                   │
└─────────────────────────────────────────────────────┘
```

- 가로 스크롤 처리: `overflow-x: auto`, `scrollbar-width: none` (webkit도 동일)
- 스크롤 힌트: 우측 끝에 페이드아웃 그라디언트 (마스크 이미지 기법) 처리
- 칩 최소 너비: `white-space: nowrap`으로 줄바꿈 방지

## 3. 점진적 공개 (Progressive Disclosure) 흐름

### 3-1. 5단계 진입 흐름 전체 맵

```
[1단계: 첫 진입]
     ↓ 섹터 바에서 섹터 탭
[2단계: 섹터 선택]
     ↓ 그래프에서 노드 클릭
[3단계: 종목 클릭 (center 전환)]
     ↓ 관계 칩으로 엣지 유형 토글
[4단계: 이웃 탐색 (관계 필터)]
     ↓ 관계 카드 패널에서 "Deep" 클릭
[5단계: Deep Dive (전용 워크스페이스)]
```

### 3-2. 단계별 표시 규칙

#### 1단계: 첫 진입 (아무것도 선택 안 됨)

```
표시 요소:
- 섹터 바: 전체 표시 (기존과 동일)
- 그래프 영역: 빈 상태 디자인 (섹션 4 참조)
- 관계 칩 바: 비활성 (dim 처리, 클릭 불가)
- 관계 카드 패널: "섹터를 선택하면 오늘의 주목 종목이 나타납니다" 텍스트

노드: 없음
엣지: 없음
라벨: 없음
툴팁: 없음
```

#### 2단계: 섹터 선택

```
표시 요소:
- 선택 섹터 칩 하이라이트 (파랑 테두리)
- 그래프: 해당 섹터 시드 노드 최대 20개 등장
  - 등장 애니메이션: 200ms fade-in + scale 0.6 → 1.0
- 관계 칩 바: 활성화 (ON 상태로 켜짐)
- 관계 카드 패널: 시드 카드 리스트 (기존 SeedCardList 유지)

노드: 섹터 내 시드 (seed_type별 색상 원형)
     라벨: ticker 심볼 (항상 표시, 최대 6자)
엣지: 시드 간 관계 (ON 칩 필터 적용)
     굵기: truth_score / 40 → 최소 1px, 최대 4px
라벨: 없음 (엣지 라벨은 호버 시에만)
툴팁: 없음
```

#### 3단계: 종목 클릭 (center 전환)

```
트리거: 그래프 노드 클릭 또는 시드 카드의 "여기서 탐색" 버튼

표시 요소:
- 클릭된 노드: 28px center로 전환, 흰색 외곽선 3px
- 나머지 노드: 0.3 불투명도로 fade
- center 주변에 1차 이웃 등장: Ring 1 (반경 160px) 시멘틱 배치
  - 등장 애니메이션: 300ms, 각도 방향에서 slide-in 효과
- 관계 카드 패널: RelationCardGroups 표시 (기존 유지)

노드: center (28px) + Ring 1 이웃 (14~20px)
     center 라벨: ticker + company_name 두 줄
     Ring 1 라벨: ticker만 (항상 표시)
엣지: ON 칩에 해당하는 타입만, 관계 타입별 색상·굵기·선 스타일
툴팁 (노드 호버):
  ┌──────────────────────────┐
  │ TSMC                     │
  │ Taiwan Semiconductor     │
  │ 관계: CUSTOMER_OF (공급망)│
  │ 신뢰도: 87               │
  │ 오늘: +1.2%              │
  └──────────────────────────┘
  - 위치: 노드 우측 12px offset
  - 등장: 150ms, 사라짐: 200ms
  - 모바일: 탭 후 0.5초 delay로 표시 (의도치 않은 트리거 방지)
```

#### 4단계: 이웃 탐색 (관계 필터)

```
트리거: 관계 칩 토글

표시 요소:
- ON 칩 대응 엣지: 불투명도 0.85, 색상 풀
- OFF 칩 대응 엣지: 완전 숨김 (display:none 수준)
- OFF 칩과만 연결된 노드: 불투명도 0.2로 dim
- 관계 카드 패널: ON 관계 타입 그룹만 표시 (나머지 접힘)

노드: center + ON 관계 연결 노드 (불투명), OFF만 연결 노드 (dim)
엣지: ON 필터 통과한 엣지만 완전 표시

## 6. 색상 시스템 재정리

현재 graphStyles.ts의 색상 체계를 유지하되, 사용 색상 수를 줄이는 통합안:

### 6-1. 엣지 색상 계층

```
1계층 — 구조적 비즈니스 관계 (실선, 두꺼움):
  SUPPLIES_TO, CUSTOMER_OF → #F97316 (오렌지)  굵기 3px
  COMPETES_WITH             → #EF4444 (빨강)   굵기 2.5px

2계층 — 구조적 비교 관계 (실선, 보통):
  PEER_OF                   → #3B82F6 (파랑)   굵기 2px

3계층 — 시장 신호 관계 (점선, 얇음):
  CO_MENTIONED              → #A855F7 (보라)   굵기 1.5px dash [5,4]
  HAS_THEME                 → #14B8A6 (틸)     굵기 1px   dash [8,4]

**Why 색상 계층화**: 투자 의사결정에서 공급망·경쟁은 펀더멘털 정보(1계층),

자연스럽게 구분된다. 색상 두께와 선 스타일이 이 계층을 강화한다.

### 6-2. 노드 색상 (섹터)

graphStyles.ts의 SECTOR_COLORS 10종 유지.
단, 동일 섹터 내 모든 노드가 같은 색이 되어 구분이 어려우므로

  ctx.beginPath();
  ctx.arc(x, y, 28, 0, Math.PI * 2);
  ctx.fillStyle = getSectorColor(node.sector);
  ctx.fill();
  ctx.shadowBlur = 0;
  // 흰색 외곽선

## 8. 구현 시 주의사항

### 8-1. react-force-graph-2d의 fx/fy 고정 한계

react-force-graph-2d는 노드 객체에 `fx`/`fy`를 설정하면 d3가 해당 노드를
위치 고정으로 처리한다. 그러나 **데이터 갱신 시 graphData 객체를 새로 생성하면
d3 시뮬레이션이 리셋되어 fx/fy가 초기화되는 문제**가 있다.

우회법:
- `graphData`를 `useMemo`로 메모이제이션하되, 노드 id 배열이 변경될 때만 재계산.
- 또는 `nodePositions` ref를 별도로 관리하고, 노드 렌더링 시 ref에서 fx/fy를 덮어쓴다.
- 가장 안정적인 방법: `onEngineStop` 콜백 직후 한 번만 fx/fy를 주입하고 `d3ReheatSimulation`을 호출하지 않는다.

### 8-2. Canvas 라벨 렌더링과 retina 디스플레이

`react-force-graph-2d`의 Canvas는 `devicePixelRatio`를 자동 처리하지 않는 버전이 있다.
Retina(2x) 디스플레이에서 라벨이 흐리게 보이는 경우:

```
// ForceGraph2D에 canvasProps 또는 직접 Canvas 접근 시
canvas.width = containerWidth * window.devicePixelRatio;
canvas.height = containerHeight * window.devicePixelRatio;
ctx.scale(window.devicePixelRatio, window.devicePixelRatio);
```

현재 GraphCanvas.tsx의 `nodeCanvasObject`에서 폰트 크기를 픽셀 단위로 지정하므로
devicePixelRatio 적용 여부를 확인하고, 라이브러리가 처리하지 않으면 수동 scale을 적용한다.
