## 1. 제품 정체성

### 1-1. Portfolio Coach는 무엇인가

Portfolio Coach는 **설명 가능한 투자 코치**다. 분석 플랫폼이 아니다.

사용자에게 숫자를 나열하는 것이 아니라, "당신의 포트폴리오가 당신의 투자 철학과 얼마나 일치하는지"를 진단하고, 불일치가 있을 때 그 이유를 설명하며, 개선 방향을 제시하는 제품이다.

핵심 질문: **"내가 실제로 하고 있는 투자가, 내가 하고 싶은 투자와 맞는가?"**

### 1-2. 프리셋의 역할

프리셋은 추천 알고리즘이 아니다. **투자 이론의 렌즈**다.

같은 포트폴리오라도 GARP 렌즈로 보면 "성장 대비 가격이 비싸다"는 이야기가 나오고, Buffett 렌즈로 보면 "경쟁우위는 좋지만 이익의 일관성이 부족하다"는 이야기가 나온다. Coach는 사용자가 선택한 렌즈의 관점에서 솔직하게 말해주는 역할이다.

### 1-3. 역할 경계

| 구성요소 | 역할 | 허용/비허용 |
|---|---|---|
| Coach | 포트폴리오 구조 진단 | 허용: 지표별 강약점, 비교 기준 대비 위치, 구조적 문제 지적 |
| Coach | | 비허용: "이 종목을 사라", "이 종목을 팔아라" 류의 직접 추천 |
| Screener | 종목 필터링 | 사용자가 프리셋 조건으로 직접 검색하는 도구. Coach와 책임 분리 |
| Chain Sight | 관계 탐색 | 종목 간 공급망·경쟁·협력 관계 시각화. 사용할수록 관계 맵 확장 |
| Thesis Control | 투자 가설 관리 | 개별 종목의 투자 근거 설정·추적·검증 |

## 2. 사용자 경험 구조

### 2-1. 투 트랙 개요

| | 트랙 A: 자산 현황 | 트랙 B: 코칭 분석 |
|---|---|---|
| 목적 | 내 돈이 지금 어디에 얼마나 있는가 | 내 투자가 내 철학과 맞는가 |
| 데이터 | 종목, 수량, 매수가, 평가금액, 수익률, 배당 | 프리셋 지표, 산업 대비 퍼센타일, LLM 코멘트 |
| 프리셋 관여 | 없음 (순수 팩트) | 핵심 (프리셋이 분석 관점 결정) |
| 시간축 | 자산 추이, 변동 기록, 세금 | 진단 히스토리, 사후 분석, 패턴 감지 |

### 2-2. 코칭 분석의 1차 경험: 보기모드 전환

사용자의 기본 경험은 **"테이블 생성"이 아니라 "보기모드 전환"**이다.

```
[내 포트폴리오 화면]
    │
    ├── 현황 보기 (트랙 A — 기본)
    │     수익률, 비중, 자산 추이
    │
    └── 코칭 보기 (트랙 B)
          ┌── 프리셋 선택 드롭다운: [GARP ▼]
          │
          └── 선택한 프리셋 관점으로 즉시 분석 결과 표시
              (저장하지 않아도 볼 수 있음)
```

- 사용자는 프리셋을 바꿔가며 같은 포트폴리오를 다른 관점으로 볼 수 있음
- 기본적으로 **일회성 보기**. 매번 현재 데이터로 실시간 분석
- "이 분석 결과를 저장하고 싶다" → Saved Analysis로 스냅샷 저장

### 2-3. Saved Analysis (저장된 분석)

| 항목 | 내용 |
|---|---|
| 생성 시점 | 사용자가 "이 분석 저장" 버튼을 누를 때 |
| 저장 내용 | 분석 시점, 적용 프리셋, 종목 목록, 각 지표 퍼센타일, LLM 코멘트, 진단 카드 |
| 용도 | 시간 경과에 따른 비교, 프리셋 변경 전후 비교, 월간 복기 리포트 |
| 불변성 | 저장된 시점의 스냅샷. 이후 데이터 변동에 영향받지 않음 |

### 2-4. 파워유저 확장: 분석 테이블 (Phase 2)

보기모드 전환만으로는 부족한 파워유저를 위해, 별도 분석 테이블을 만들 수 있는 기능을 Phase 2에서 제공.

- 보유 종목 외에 예비 후보, 관심 종목을 자유롭게 추가
- 종목 복사(같은 종목을 다른 프리셋 테이블에서 분석)
- 여러 테이블을 동시에 운영하며 관점별 비교

이 기능은 MVP에서는 제공하지 않으며, 1차 경험(보기모드 전환)이 충분히 검증된 후 추가한다.

## 3. 분석 엔진

### 3-1. 퍼센타일 기반 진단 (핵심)

분석의 핵심 출력값은 **산업/섹터 대비 퍼센타일(percentile)**이다.

```
NVDA의 ROE: 시장 전체 상위 8% → 반도체 산업 내 상위 12%
                                    ↑ 이것이 핵심 출력값
```

| 항목 | 설계 |
|---|---|
| 기본 비교군 | 동일 산업(industry) 내 종목 |
| fallback 규칙 | industry 표본 < 15개 → sector로 확장 → sector도 부족 시 broader universe |
| 출력 형식 | 0~100 퍼센타일 (연속값) |
| 방향성 | 지표별로 "높을수록 좋음" 또는 "낮을수록 좋음" 정의 |
| 갱신 주기 | EOD 배치 (1일 1회) |

**왜 산업 대비인가:**
- P/E 30이 소프트웨어 업계에서는 평균이지만 유틸리티에서는 극도로 비쌈
- 시장 전체 대비 비교는 이런 산업 특성을 무시하여 오해를 유발
- 사용자에게 "당신 종목이 같은 업종 내에서 어디에 있는지"를 알려주는 것이 더 실용적

### 3-2. 레벨 태그 (보조 표현)

퍼센타일을 사용자에게 직접 보여주면 직관성이 떨어지므로, 시각적 레벨링 용도로 5단계 태그를 부여한다.

| 레벨 | 퍼센타일 범위 | 시각 표현 | 의미 |
|---|---|---|---|
| 5 | 상위 0~20% | 🟢🟢 | 업종 내 최상위권 |
| 4 | 상위 20~40% | 🟢 | 업종 내 우수 |
| 3 | 중간 40~60% | 🟡 | 업종 평균 수준 |
| 2 | 하위 20~40% | 🔴 | 업종 내 하위권 |
| 1 | 하위 0~20% | 🔴🔴 | 업종 내 최하위권 |

**용도:** 지표 정렬 순서 결정, UI 색상 표시, 한눈에 파악하기 위한 보조. 분석 엔진의 핵심은 아님.

### 3-3. 지표 정렬 순서

테이블 내 종목들의 레벨 태그를 합산하여 정렬. 강점(합산 높음) → 약점(합산 낮음) 순서.

| 항목 | 로직 |
|---|---|
| 1차 정렬 | 레벨 태그 합산 높은 순 → 낮은 순 |
| 타이브레이커 | 레벨 5 개수 → 4 개수 → 3 개수 → 2 개수 → 1 개수 순 |

이 정렬은 "내 포트폴리오가 업종 대비 어디가 강하고 어디가 약한지"의 순서를 자동으로 결정한다.

## 4. 프리셋 구조

### 4-1. 2단계 구조

```
카테고리 (8~9개)
  └── 세부 프리셋 (카테고리당 약 3개, 총 16~25개)
```

### 4-2. 온보딩: 점진적 공개 (Progressive Disclosure)

프리셋 수 자체를 줄이는 것이 아니라, **사용자가 보는 복잡도를 단계적으로 조절**한다.

**1단계 (첫 진입):**
- "어떤 스타일의 투자를 하고 계신가요?" → 카테고리 4~5개만 노출
- 예: 안정적으로 / 성장 중심으로 / 배당 중심으로 / 균형 있게 / 잘 모르겠음
- 선택하면 해당 카테고리의 디폴트 프리셋 자동 적용

**2단계 (코칭 보기 진입 후):**
- 프리셋 드롭다운에서 현재 카테고리의 세부 프리셋 2~3개 노출
- "GARP"가 아니라 "성장주를 합리적 가격에 — GARP" 같은 설명형 레이블

**3단계 (설정 또는 탐색):**
- 전체 카테고리 + 전체 프리셋 목록 공개
- 각 프리셋의 철학, 사용 지표, 사용자 설명문 제공

이 구조에서 프론트엔드/UX가 해결하는 부분과 구조적으로 줄여야 하는 부분은 다음과 같다:

| 구분 | 내용 |
|---|---|
| UX로 해결 | 노출 순서/개수 조절, 설명형 레이블, 카테고리 → 세부 프리셋 단계적 탐색 |
| 구조적으로 결정 필요 | 카테고리당 프리셋 수 (3개 이하 권장), 프리셋 간 지표 중복도 관리, 프리셋 간 진단 차별화 검증 |

### 4-3. 카테고리 목록

| # | 카테고리 | 코칭 질문 |
|---|---|---|
| 1 | 가치 투자 | "이 가격이 이 기업의 진짜 가치 대비 합리적인가?" |
| 2 | 성장 투자 | "이 기업의 성장이 지속 가능하고, 그 대가로 지불하는 가격이 적정한가?" |
| 3 | 배당/인컴 | "이 기업이 앞으로도 배당을 올릴 수 있는 체력이 있는가?" |
| 4 | 모멘텀/추세 | "시장의 흐름이 이 종목에 우호적인가?" |
| 5 | 퀄리티/방어 | "시장이 흔들릴 때 이 포트폴리오가 버틸 수 있는가?" |
| 6 | 퀀트/팩터 | "어떤 수익 요인(팩터)에 노출되어 있는가?" |
| 7 | 역발상 | "시장이 과잉 반응한 곳에서 기회를 잡고 있는가?" |
| 8 | 테마/전략 | "특정 메가트렌드에 얼마나 효과적으로 노출되어 있는가?" |

> 세부 프리셋 선정은 Step 1에서 수행. 선정 기준: Stock-Vis 데이터 구현 가능성, 프리셋 간 차별화, 사용자 수요.

## 5. 지표 우선순위 계층

### 5-1. 왜 계층이 필요한가

가중치 매트릭스는 폐기했지만, 모든 지표가 동일한 중요도를 갖는 것은 아니다. GARP에서 PEG는 핵심 철학 그 자체지만, 부채비율은 참고 수준이다. 이 차이를 반영하지 않으면:

- LLM 코멘트가 핵심과 참고를 동일한 비중으로 언급
- 진단 카드가 사소한 지표의 약점을 주요 문제로 부각
- 사용자가 "이 프리셋에서 정말 중요한 게 뭔데?"를 파악하기 어려움

### 5-2. 3계층 구조

| 계층 | 역할 | 프리셋당 개수 | 예시 (GARP) |
|---|---|---|---|
| **Core (핵심)** | 이 프리셋의 철학을 정의하는 지표. 여기가 나쁘면 진단 카드의 주요 대상 | 3~5개 | PEG, EPS 성장률, P/E |
| **Supporting (보조)** | 핵심 지표를 뒷받침하거나 보완하는 지표. 코멘트에 포함되지만 카드 대상은 Core 이후 | 3~5개 | ROIC, 부채비율, 매출 성장 지속성 |
| **Context (참고)** | 맥락을 제공하지만 이 프리셋의 핵심 판단과는 거리가 있는 지표. 펼치기로 확인 가능 | 2~5개 | Beta, 배당수익률, 시가총액 |

### 5-3. 계층이 반영되는 곳

**진단 카드 생성:**
- Core 지표 중 약점이 있으면 최우선으로 카드 생성
- Core에 문제 없으면 Supporting으로 내려감
- Context 지표는 단독으로 카드를 생성하지 않음 (다른 카드의 맥락 보충으로만 활용)

**LLM 코멘트 톤:**
- Core: "PEG가 업종 하위 30%입니다. GARP 관점에서 성장 대비 가격이 높다는 의미입니다."
- Supporting: "부채비율이 업종 평균 수준입니다. 현재로서는 양호하나 모니터링이 필요합니다."
- Context: 별도 코멘트 없이 수치만 표시. 사용자가 궁금하면 펼쳐서 확인

**UI 노출:**
- Core + Supporting: 기본 화면에 항상 표시
- Context: "더 보기" 뒤에 숨김. 관심 있는 사용자만 확인

### 5-4. 프리셋 차별화가 계층으로 드러남

같은 지표(예: P/E)라도 프리셋에 따라 계층이 달라진다.

| 지표 | GARP | Buffett Quality Value | Dividend Growth |
|---|---|---|---|
| PEG | **Core** | Supporting | Context |
| ROE | Supporting | **Core** | Supporting |
| 배당성장률 | Context | Context | **Core** |
| P/E | **Core** | Supporting | Context |
| ROIC 지속성 | Supporting | **Core** | Context |
| 배당성향 | Context | Context | **Core** |

사용자가 프리셋을 바꾸면 같은 종목이라도 강조되는 지표가 바뀌고, 진단 카드의 내용이 달라진다. 이것이 "같은 데이터, 다른 렌즈, 다른 코칭"의 핵심 메커니즘이다.

## 6. Coach 분석 계층

### 6-1. 왜 계층화하는가

Portfolio Coach는 Chain Sight, Thesis Control, Dashboard 매크로 데이터와 연결되면 강력해진다. 그러나 초기부터 이 모든 모듈에 의존하면:
- 다른 모듈 미완성 시 Coach도 작동 불가
- 사용자가 Chain Sight를 안 쓰면 Coach 가치가 급감
- 독립 제품으로서의 첫 가치 제공이 불가능

### 6-2. Layer 1: 독립 분석 (MVP)

Chain Sight, Thesis Control, 매크로 데이터 없이 FMP 재무/가격 데이터만으로 작동.

| 제공 기능 | 데이터 소스 |
|---|---|
| 프리셋별 지표 진단 | FMP 재무제표 + 가격 |
| 산업 대비 퍼센타일 | FMP 배치 데이터 |
| 지표별 LLM 코멘트 | 퍼센타일 + 프리셋 적정 기준 |
| 진단 카드 3개 | Core 지표 약점 기반 |
| 강점/약점 정렬 | 레벨 태그 합산 |

**MVP에서 사용자가 얻는 가치:**
- "내 포트폴리오가 GARP 관점에서 어떤 상태인지" 즉시 확인
- 업종 대비 강한 점/약한 점 파악
- 프리셋을 바꿔가며 다른 관점으로 같은 포트폴리오 진단

### 6-3. Layer 2: 연결 강화 분석 (Phase 2)

다른 모듈과 연결하여 Layer 1에서 볼 수 없는 인사이트 추가.

| 연결 모듈 | 추가 분석 |
|---|---|
| Chain Sight | 관계 클러스터 기반 실질 분산도, 공급망 중복 리스크, HRP 적용 |
| Thesis Control | thesis 건강도, 전제 유효성, Black-Litterman 확신도 기반 비중 적정성 |
| Dashboard 매크로 | 매크로 환경 적합도, 금리 민감도, 경기방어/민감 비율 |
| History | BPT 행동 편향 분석 (처분 효과, 조기 익절 패턴 등) |

**Layer 2가 주는 추가 가치:**
- "종목은 7개지만 Chain Sight 기준 실질 클러스터가 2개뿐입니다" (Layer 1에서는 불가능)
- "thesis 확신도 대비 비중이 과도합니다" (Thesis Control 필요)
- "고금리 환경에서 이 포트폴리오의 75%가 역풍 구간입니다" (매크로 데이터 필요)

### 6-4. Layer 간 UX 처리

Layer 2 데이터가 없을 때:
- 해당 지표는 "데이터 없음"으로 표시하되, 에러가 아닌 **업그레이드 유도**로 처리
- "Chain Sight에서 이 종목의 관계를 탐색하면, 숨겨진 집중 리스크를 확인할 수 있습니다"
- Layer 1만으로도 완결된 분석 경험 제공. Layer 2는 추가 깊이

## 7. 진단 출력 구조

### 7-1. 메인 출력: 종합 점수 대신 상태 요약

종합 100점 점수를 메인 KPI로 사용하지 않는다. 사용자가 "72점이면 뭘 해야 하는데?"를 이해하기 어렵기 때문이다.

**메인 출력 구조:**

```
┌─────────────────────────────────────────────┐
│ 한 줄 상태 요약 (LLM 생성)                    │
│ "GARP 관점에서 성장성은 우수하지만,            │
│  밸류에이션이 업종 대비 높은 편입니다."          │
├─────────────────────────────────────────────┤
│ 강점 3개              │ 약점 3개              │
│ 🟢 ROIC 업종 상위 12% │ 🔴 PEG 업종 하위 30%  │
│ 🟢 EPS성장 상위 15%   │ 🔴 부채비율 하위 35%   │
│ 🟢 매출성장 상위 20%  │ 🟡 thesis 2개 미설정   │
├─────────────────────────────────────────────┤
│ 진단 카드 3개 (Core 약점 기반)                 │
│ [카드 1] [카드 2] [카드 3]                     │
├─────────────────────────────────────────────┤
│ 지표 상세 (레벨 합산 순 정렬)                   │
│ Core → Supporting → Context 순                │
│ 각 지표: 퍼센타일 + 종목별 분포 + LLM 코멘트    │
└─────────────────────────────────────────────┘
```

### 7-2. 종합 점수의 위치

종합 점수는 메인 기능이 아닌 **상태 배지** 수준으로만 존재.

| 용도 | 위치 |
|---|---|
| 시간 추이 비교 | Saved Analysis 간 비교 화면에서만 사용 |
| 프리셋 변경 전후 | "GARP 68 → Quality Value 74" 수준의 요약에서만 사용 |
| 메인 화면 | 표시하지 않음 또는 작은 배지로만 표시 |

산출 방식: Core 지표 퍼센타일의 가중 평균 또는 레벨 합산의 정규화. 상세 방식은 추후 결정.

### 7-3. 진단 카드 구조

단순 경고가 아니라, **설명 책임이 있는 카드**로 설계한다.

각 카드는 다음 4개 요소를 반드시 포함:

| 요소 | 역할 | 예시 |
|---|---|---|
| 무엇이 문제인가 | 팩트 진술 | "PEG 평균이 업종 하위 30%에 위치합니다." |
| 어떤 기준에서 문제인가 | 비교 기준 명시 | "반도체 산업 87개사 대비 퍼센타일 기준." |
| 왜 이 프리셋에서 중요한가 | 프리셋 철학 연결 | "GARP은 성장 대비 합리적 가격을 핵심으로 봅니다. PEG가 높다는 것은 성장이 가격에 이미 반영되었을 가능성을 의미합니다." |
| 예외/트레이드오프 | 맥락 보충 | "다만 NVDA의 경우 AI 인프라 독점적 위치를 고려하면 프리미엄이 정당화될 수 있습니다. thesis에서 이 관점을 검증해보시기 바랍니다." |

**문체 원칙:**
- 단정형("비싸다") 보다 조건형/설명형("업종 대비 높은 편이며, 성장 둔화 시 하방 리스크가 커질 수 있습니다")
- Coach는 판단을 내리는 것이 아니라, 판단에 필요한 정보를 제공하는 역할

### 7-4. 지표 상세

레벨 합산 순으로 정렬. 각 지표마다:

```
[지표명] — [계층 태그: Core/Supporting/Context]
퍼센타일: 업종 내 상위 28% (비교군: 반도체 산업 87개사)

종목별 분포:
  🟢🟢 레벨5: NVDA (상위 8%), MSFT (상위 12%)
  🟢   레벨4: AMZN (상위 25%)
  🟡   레벨3: LLY (상위 45%)
  🔴   레벨2: INTC (하위 35%)

[LLM 한 줄 코멘트]
"5개 종목 중 3개가 업종 상위 25% 이내. 다만 INTC의 수익성이 하락세."
```

## 8. 순환 루프

### 8-1. 플로우

```
Portfolio Coach 진단 → 약점 발견
  → "이 약점을 보완할 종목을 찾아볼까요?" (행동 유도)
  → Screener (프리셋 기준으로 필터링 — 사용자 주도)
  → Chain Sight (후보 종목의 관계 탐색 — 새로운 관계 노드 생성)
  → Thesis Control (투자 가설 설정)
  → Portfolio 예비 후보 등록
  → Coach 재진단 ("이 종목 추가 시 변화 분석")
  → (루프 반복)
```

### 8-2. 락인 메커니즘

Coach → Screener → Chain Sight 루프를 반복할수록:
- Chain Sight의 관계 맵이 사용자별로 고유하게 확장
- Thesis Control의 투자 가설 히스토리가 축적
- Saved Analysis의 진단 히스토리가 쌓여 사후 분석 가치 증가

이 데이터는 사용자가 다른 서비스로 이동하면 처음부터 다시 구축해야 하므로, 사용 기간에 비례하여 이탈 비용이 증가한다.

## 9. 데이터 설계

### 9-1. 퍼센타일 배치 현실화

**초기 유니버스 제한 전략:**

| Phase | 유니버스 | 산출 방식 |
|---|---|---|
| MVP | S&P 500 + 사용자 보유 종목의 산업 peers | Core 지표만 사전 배치 계산 |
| Phase 2 | Russell 3000 | Core + Supporting 사전 배치 |
| Phase 3 | 글로벌 확장 | 전체 사전 배치 + 캐시 |

**온디맨드 계산:**
- 사용자가 S&P 500 외 종목을 입력한 경우, 해당 종목의 산업 peers를 실시간으로 조회하여 퍼센타일 산출
- 결과는 캐시하여 동일 종목/지표 재요청 시 재계산 방지
- 캐시 TTL: 24시간 (EOD 배치와 동기화)

### 9-2. 버전 관리

과거 분석 결과와 현재 결과의 비교 가능성을 위해 다음 항목에 버전을 부여한다.

| 버전 항목 | 설명 | 변경 빈도 |
|---|---|---|
| `preset_version` | 프리셋의 지표 구성, 계층, 적정 범위가 변경될 때 | 분기 1회 이하 |
| `metric_definition_version` | 지표의 산식, 방향성, 비교군 규칙이 변경될 때 | 분기 1회 이하 |
| `scoring_version` | 레벨 태그 기준(퍼센타일 구간) 변경 시 | 연 1회 이하 |
| `prompt_version` | LLM 코멘트/카드 생성 프롬프트 변경 시 | 수시 (A/B 테스트 포함) |
| `universe_version` | 퍼센타일 산출 대상 유니버스 변경 시 | Phase 전환 시 |

**Saved Analysis에는 분석 시점의 모든 버전을 함께 저장**하여, 이후 "같은 조건에서의 비교"와 "조건 변경에 의한 차이"를 구분할 수 있도록 한다.

### 9-3. 결측치 정책 (다음 단계 최우선 과제)

지표를 계산할 수 없는 경우의 분류:

| 상태 | 의미 | UI 처리 |
|---|---|---|
| Missing | 데이터 소스에 값이 없음 | "데이터 없음" 표시. 해당 지표 정렬에서 제외 |
| Not Applicable | 해당 지표가 이 종목에 적용되지 않음 (예: 무배당 기업의 배당성향) | "해당 없음" 표시. 별도 설명 제공 |
| Delayed | 데이터가 아직 갱신되지 않음 (실적 발표 직후 등) | 이전 값 표시 + "갱신 대기" 태그 |
| Insufficient History | 상장 기간이 짧아 5년 데이터 부족 등 | 가용 기간으로 계산 + "제한적 데이터" 태그 |
| Temporarily Unstable | 일시적 이상치 (인수합병, 대규모 상각 등으로 지표 왜곡) | 값 표시 + "일시적 왜곡 가능" 태그 |

**결측 종목의 퍼센타일 처리:**
- 해당 종목은 해당 지표의 레벨 합산에서 제외
- 만점 기준을 (참여 종목 수 × 5)로 조정하여 합산 비율로 비교

### 9-4. Metric Dictionary 구조 (다음 단계에서 작성)

각 지표에 대해 정의할 필드:

| 필드 | 설명 |
|---|---|
| `metric_id` | 고유 식별자 (예: `pe_ratio`, `roe_ttm`) |
| `display_name` | 사용자에게 표시되는 이름 |
| `description` | 지표 설명 (1~2문장) |
| `formula` | 산출 공식 |
| `data_source` | FMP 필드명 또는 계산 로직 |
| `direction` | `higher_is_better` 또는 `lower_is_better` |
| `comparison_group` | 기본 비교군 (industry / sector / universe) |
| `fallback_rule` | 비교군 표본 부족 시 fallback 규칙 |
| `refresh_frequency` | 갱신 주기 (EOD / 분기 / 연간) |
| `missing_policy` | 결측 시 처리 방법 |
| `display_condition` | 표시 가능 조건 (예: 배당 지표는 배당 지급 기업에만) |
| `applicable_presets` | 이 지표를 사용하는 프리셋 목록 |
| `tier_per_preset` | 프리셋별 계층 (Core/Supporting/Context) |
