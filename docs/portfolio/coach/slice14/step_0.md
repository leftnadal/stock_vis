- Slice 14 = 게이트 보정 전용 슬라이스. Slice 13 Step 0a에서 ADDITIVE로

  (warn_below = fail_below × 1.5 류 임시 공식)다. 이를 실제 시장 데이터
  기반 실측 경계값으로 교체하는 것이 슬라이스 본체(Part 1~3).
- 본 Step 0는 그 본체에 들어가기 전 단계: ①진입 전 사실 확인,
  ②베이스라인 고정, ③#63 cost ledger 신설.

1. Step 0는 게이트 점수 경로·gate_tiers 평가 로직을 한 줄도 수정하지
   않는다. 경계값 교체는 Part 2의 일이다.

   동작(허용/거부 판정)을 일절 변경하지 않는다. 비용 한도 초과 시 호출을

   ledger는 "무엇이 일어났는지 기록"만 하고 "막는" 행위는 하지 않는다.

   해시 불변이어야 한다. 깨지면 즉시 중단·보고.
4. 메모리/지시서의 추정은 작업 1에서 코드로 먼저 검증한다. 추정이 코드와

■ 작업 2 — #63 cost ledger 신설 (분리 커밋, 기록 전용)
목적: LLM 호출별 비용을 파일에 append해 누적 비용을 실측값으로 만든다.
현재 누적 비용 수치는 closing 보고서 기재값일 뿐 미검증 상태다.
───────────────────────────────────────────────────────────────
[설계 원칙 — 위반 금지]

- ledger는 append-only 기록 장치다. 호출 허용/거부 판정에 일절 관여하지
  않는다 (그 작업 = #64, 이번 범위 아님).
- 기존 CostGuard의 공개 인터페이스·차단 로직·reset_for_slice 동작을
  수정하지 않는다. ledger 기록은 비용이 집계되는 지점에 "추가"만 한다.
- 작업 1의 1-4 결과와 충돌하면(예: 이미 ledger 비슷한 것이 존재) 진행을
  멈추고 보고 — 추정대로 강행하지 말 것.

2-1. ledger 파일 포맷 결정 및 경로

- 형식: 1행 = LLM 호출 1건. 권장 컬럼:
  timestamp, slice, entry_point(e1~e6 등), provider, model,
  input_tokens, output_tokens, cost_usd, fallback_from.
- 저장 형식은 append 안전성이 높은 JSONL(줄 단위 JSON) 권장.
  프로젝트에 이미 선호 포맷이 있으면(작업 1-4에서 확인) 그에 맞춤.
- 경로는 리포지토리 정책에 맞추되, 작업 1-4에서 cost 관련 파일들이
  있는 위치를 따른다. 경로를 보고에 명기.

2-2. ledger 기록 로직 추가

- LLM 호출 비용이 집계되는 지점(작업 1-4에서 특정한 곳)에 ledger
  append 호출을 1곳 추가. 기존 집계 로직은 수정하지 않고 호출만 끼운다.
- append 실패가 본 호출 흐름을 깨지 않도록 방어(기록 실패는 로깅만,
  예외 전파 금지). ledger는 보조 장치이지 호출 경로의 의존성이 아니다.
- 동시성·파일 락은 과설계하지 않는다. 이 프로젝트는 슬라이스 단위
  순차 실행이므로 단순 append로 충분.

2-3. 테스트 (회귀 +예상 6~12, ±30% 임계 +4~16)

- ledger append 1건 → 파일에 정확한 컬럼/값으로 1행 추가 확인.
- 연속 호출 → 행이 누적(append)되고 기존 행이 보존됨 확인.
- append 실패 상황 모킹 → 본 호출 흐름이 예외 없이 정상 완료됨 확인
  (ledger 보조성 보증).
- CostGuard의 기존 차단 로직·카운트·reset_for_slice 동작이 ledger
  추가 전후로 동일함을 확인 (#64와 무관함을 회귀로 못박기).
- 누적 비용 합산 유틸이 있다면(또는 신설 시) ledger 파일을 읽어
  cost_usd 합을 반환하는 함수 1개 + 그 테스트.

2-4. COST_POLICY.md 갱신

- ledger 신설 사실, 파일 경로·포맷, "기록 전용(차단은 #64 별도)"
  원칙을 COST_POLICY.md에 짧게 1개 섹션으로 추가.
