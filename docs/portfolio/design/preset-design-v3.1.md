## 1. 제품 정체성

### 1-1. Portfolio Coach는 무엇인가

Portfolio Coach는 **설명 가능한 투자 코치**다. 분석 플랫폼이 아니다.

사용자에게 숫자를 나열하는 것이 아니라, "당신의 포트폴리오가 당신의 투자 철학과 얼마나 일치하는지"를 진단하고, 불일치가 있을 때 그 이유를 설명하며, 개선 방향을 제시하는 제품이다.

핵심 질문: **"내가 실제로 하고 있는 투자가, 내가 하고 싶은 투자와 맞는가?"**

### 1-2. 프리셋의 역할

프리셋은 추천 알고리즘이 아니다. **투자 이론의 렌즈**다.

같은 포트폴리오라도 GARP 렌즈로 보면 "성장 대비 가격이 비싸다"는 이야기가 나오고, Buffett 렌즈로 보면 "경쟁우위는 좋지만 이익의 일관성이 부족하다"는 이야기가 나온다. Coach는 사용자가 선택한 렌즈의 관점에서 솔직하게 말해주는 역할이다.

### 1-3. 역할 경계

| 구성요소 | 역할 | 허용/비허용 |
|---|---|---|
| Coach | 포트폴리오 구조 진단 | 허용: 지표별 강약점, 비교 기준 대비 위치, 구조적 문제 지적 |
| Coach | | 비허용: "이 종목을 사라", "이 종목을 팔아라" 류의 직접 추천 |
| Screener | 종목 필터링 (Phase 2) | 사용자가 프리셋 조건으로 직접 검색하는 도구. Coach와 책임 분리 |
| Chain Sight | 관계 탐색 (Phase 2) | 종목 간 공급망·경쟁·협력 관계 시각화. 사용할수록 관계 맵 확장 |
| Thesis Control | 투자 가설 관리 (Phase 2) | 개별 종목의 투자 근거 설정·추적·검증 |

## 2. 사용자 경험 구조

### 2-1. 투 트랙 개요

| | 트랙 A: 자산 현황 | 트랙 B: 코칭 분석 |
|---|---|---|
| 목적 | 내 돈이 지금 어디에 얼마나 있는가 | 내 투자가 내 철학과 맞는가 |
| 데이터 | 종목, 수량, 매수가, 평가금액, 수익률, 배당 | 프리셋 지표, 산업 대비 퍼센타일, LLM 코멘트 |
| 프리셋 관여 | 없음 (순수 팩트) | 핵심 (프리셋이 분석 관점 결정) |
| 시간축 | 자산 추이, 변동 기록, 세금 | Saved Analysis 히스토리, 사후 분석 |

### 2-2. 코칭 분석의 1차 경험: 보기모드 전환

사용자의 기본 경험은 **"테이블 생성"이 아니라 "보기모드 전환"**이다.

```
[내 포트폴리오 화면]
    │
    ├── 현황 보기 (트랙 A — 기본)
    │     수익률, 비중, 자산 추이
    │
    └── 코칭 보기 (트랙 B)
          ┌── 프리셋 선택 드롭다운: [GARP ▼]
          │
          └── 선택한 프리셋 관점으로 즉시 분석 결과 표시
              (저장하지 않아도 볼 수 있음)
```

- 사용자는 프리셋을 바꿔가며 같은 포트폴리오를 다른 관점으로 볼 수 있음
- 기본적으로 **일회성 보기**. 매번 현재 데이터로 실시간 분석
- 별도 저장 없이도 최근 분석은 자동 임시 보관

### 2-3. Saved Analysis (저장된 분석)

**2단계 저장 구조:**

| 구분 | 임시 저장 (자동) | 영구 저장 (명시적) |
|---|---|---|
| 생성 | 코칭 보기 전환 시 자동 | 사용자가 "이 분석 저장" 버튼 클릭 |
| 보관 수 | 최근 5~10개 (중복 제거 기준 적용) | 무제한 |
| 수명 | 새 분석이 들어오면 오래된 것부터 삭제 | 사용자가 삭제할 때까지 영구 보관 |
| 용도 | "방금 봤던 분석"으로 돌아가기 | 시간 경과 비교, 프리셋 변경 전후 비교, 월간 복기 |
| 중복 처리 | portfolio_hash + preset_id 동일 시 최신 결과로 덮어쓰기. 포트폴리오 구성이 변경되면 새로 생성 | 해당 없음 |

**저장 내용 (임시/영구 공통):**

| 항목 | 설명 |
|---|---|
| 분석 시점 | 타임스탬프 |
| 적용 프리셋 | 프리셋 ID + preset_version |
| 포트폴리오 구성 | 종목 목록 + 비중 (구성 해시 포함) |
| 원시 지표 값 | 각 종목 × 각 지표의 실제 값 |
| 퍼센타일 결과 | 각 종목 × 각 지표의 산업 대비 퍼센타일 |
| 비교군 메타데이터 | 사용된 산업 분류, 비교군 종목 수, fallback 적용 여부 |
| LLM 코멘트 | 생성된 코멘트 전문 |
| 진단 카드 | 카드 3개 전문 |
| 상태 배지 | 5단계 정합도 |
| 버전 묶음 | metric_definition_version, scoring_version, prompt_version, universe_version |

**불변성:** 저장된 분석은 이후 데이터 변동, 프리셋 변경, 비교군 변경에 영향받지 않음. 과거 진단을 현재와 비교할 때, "무엇이 바뀌어서 결과가 달라졌는지"를 구분할 수 있도록 버전 묶음을 함께 저장.

### 2-4. 파워유저 확장: 분석 테이블 (Phase 2)

보기모드 전환만으로는 부족한 파워유저를 위해, 별도 분석 테이블을 만들 수 있는 기능을 Phase 2에서 제공.

- 보유 종목 외에 예비 후보, 관심 종목을 자유롭게 추가
- 종목 복사 (같은 종목을 다른 프리셋 테이블에서 분석)
- 여러 테이블을 동시에 운영하며 관점별 비교

이 기능은 MVP에서는 제공하지 않으며, 1차 경험(보기모드 전환)이 충분히 검증된 후 추가한다.

## 3. 분석 엔진

### 3-1. 퍼센타일 기반 진단 (핵심)

분석의 핵심 출력값은 **산업/섹터 대비 퍼센타일(percentile)**이다.

```
NVDA의 ROE: 반도체 산업 내 상위 12% (비교군: 반도체 산업 87개사)
             ↑ 이것이 핵심 출력값
```

| 항목 | 설계 |
|---|---|
| 기본 비교군 | 동일 산업(industry) 내 종목 |
| fallback 규칙 | industry 표본 < 15개 → sector로 확장 → sector도 부족 시 broader universe |
| 출력 형식 | 0~100 퍼센타일 (연속값) |
| 방향성 | 지표별로 "높을수록 좋음" 또는 "낮을수록 좋음" 정의 |
| 갱신 주기 | EOD 배치 (1일 1회) |

**왜 산업 대비인가:**
- P/E 30이 소프트웨어 업계에서는 평균이지만 유틸리티에서는 극도로 비쌈
- 시장 전체 대비 비교는 산업 특성을 무시하여 오해를 유발
- 사용자에게 "당신 종목이 같은 업종 내에서 어디에 있는지"를 알려주는 것이 더 실용적

**비교군 정책 상세 (다음 단계에서 별도 문서화 필요):**
- 산업 분류 체계: FMP 자체 분류 또는 GICS (확정 필요)
- 최소 표본 수 기준: 15개 (검증 필요. 통계적 의미를 갖는 최소 표본)
- outlier winsorizing: 상하위 1% 또는 2% clip (확정 필요)
- 특수군 처리: 금융, REIT, ETF는 별도 비교군 또는 제외 (확정 필요)

### 3-2. 레벨 태그 (보조 표현)

퍼센타일을 사용자에게 직접 보여주면 직관성이 떨어지므로, 시각적 레벨링 용도로 5단계 태그를 부여한다.

| 레벨 | 퍼센타일 범위 | 의미 |
|---|---|---|
| 5 | 상위 0~20% | 업종 내 최상위권 |
| 4 | 상위 20~40% | 업종 내 우수 |
| 3 | 중간 40~60% | 업종 평균 수준 |
| 2 | 하위 20~40% | 업종 내 하위권 |
| 1 | 하위 0~20% | 업종 내 최하위권 |

**용도:** 지표 정렬 순서 결정, UI 색상 표시, 한눈에 파악하기 위한 보조. 분석 엔진의 핵심이 아니며 퍼센타일에서 파생되는 보조 표현.

### 3-3. 지표 정렬 순서

포트폴리오 내 종목들의 레벨 태그를 합산하여 정렬. 강점(합산 높음) → 약점(합산 낮음) 순서.

| 항목 | 로직 |
|---|---|
| 1차 정렬 | 레벨 태그 합산 높은 순 → 낮은 순 |
| 타이브레이커 | 레벨 5 개수 → 4 개수 → 3 개수 → 2 개수 → 1 개수 순 |

이 정렬은 "내 포트폴리오가 업종 대비 어디가 강하고 어디가 약한지"의 순서를 자동으로 결정한다.

**향후 정교화 가능:** MVP에서는 단순 레벨 합산을 사용하되, 후속 버전에서 비중 가중 평균 레벨, 분산/불균형 패널티, 극단값 패널티 등을 도입하여 정렬 정확도를 높일 수 있다.

### 3-4. 강점/약점 선정 로직

메인 출력의 강점 3개, 약점 3개, 진단 카드 3개를 선정하는 규칙.

**영향도 산식 (계층형 복합):**

3단계로 작동한다.

1단계 — 비중 가중 badness 산출 (기본 영향도):
```
impact_score = Σ (max_level - stock_level) × stock_weight
```
각 종목의 "얼마나 나쁜가(5에서 레벨을 뺀 값)"에 포트폴리오 비중을 곱해서 합산.

2단계 — 구조적 약점 판정 (보조 플래그):
```
bad_ratio = 레벨2 이하 종목 수 / 전체 종목 수
if bad_ratio ≥ 0.5 → structural_weakness = true
```
해당 지표에서 과반수 종목이 하위권이면 구조적 약점으로 태깅.

3단계 — 최종 정렬:
```
1차: Core 지표가 Supporting/Context보다 항상 우선
2차: impact_score 높은 순
3차 (동점 시): structural_weakness가 있는 지표 우선
4차 (여전히 동점 시): 최악 종목의 비중이 큰 지표 우선
```

**구조적 약점과 집중 문제의 구분:**
- 구조적 약점: 다수 종목이 일관되게 낮음 (예: 5개 중 4개가 레벨2) → "포트폴리오 전반적으로 부채 수준이 업종 대비 높습니다"
- 집중 문제: 1개 대형 포지션이 극단적으로 나쁨 (예: 비중 30%짜리 1개가 레벨1) → "NVDA의 PEG가 업종 대비 매우 높습니다. 포트폴리오 비중이 30%이므로 전체 영향이 큽니다"
- LLM 코멘트와 진단 카드에서 이 두 유형을 구분하여 톤을 달리함

**선정 규칙:**

| 순위 | 규칙 | 이유 |
|---|---|---|
| 1 | Core 지표 우선 | 프리셋 철학상 핵심인 지표가 먼저 |
| 2 | impact_score 기반 영향도 | 비중 큰 종목의 문제가 소액 종목보다 우선 |
| 3 | 구조적 약점 플래그 | 다수 종목이 일관되게 낮은 지표 우선 |
| 4 | Context 지표는 단독 선발 불가 | Context는 다른 카드의 맥락 보충으로만 활용 |

**강점 선정:** Core 지표 중 impact_score가 가장 낮은(=가장 좋은) 3개. Core가 3개 미만이면 Supporting에서 보충.
**약점 선정:** Core 지표 중 impact_score가 가장 높은(=가장 나쁜) 3개. Core가 3개 미만이면 Supporting에서 보충.
**진단 카드:** 약점 3개와 동일한 지표 기반으로 LLM이 생성. structural_weakness 여부에 따라 코멘트 톤을 달리함.

## 4. 프리셋 구조

### 4-1. 2단계 구조

```
카테고리 (8~9개)
  └── 세부 프리셋 (카테고리당 약 3개, 총 16~25개)
```

시스템은 16~25개를 수용하도록 설계하되, MVP 오픈 시에는 10~12개로 시작한다.

### 4-2. MVP 오픈 프리셋 수

| 카테고리 | MVP 프리셋 수 | 이유 |
|---|---|---|
| 가치 투자 | 2개 | 수요 높음. Buffett vs 정량 가치의 차별화 명확 |
| 성장 투자 | 2개 | 수요 높음. GARP vs 순수 성장의 차별화 명확 |
| 배당/인컴 | 2개 | 배당 성장 vs 총주주환원의 차별화 명확 |
| 퀄리티/방어 | 2개 | 퀄리티 vs 저변동성의 차별화 명확 |
| 모멘텀/추세 | 1개 | Stock-Vis 핵심 강점이 아닌 영역 |
| 퀀트/팩터 | 1개 | Multi-Factor 하나로 커버 가능 |
| 역발상 | 1개 | 니치 수요 |
| 테마/전략 | 1개 | 니치 수요 |
| **합계** | **12개** | |

프리셋 하나를 추가하는 데 필요한 작업:
- 지표 선정 + 계층 배정
- 지표별 비교 방향성 정의
- LLM 코멘트 프롬프트 설계
- 진단 카드 논리 설계
- 온보딩 노출 순서 결정
- 사용자 설명문 작성

시스템 구조는 확장 가능하게 설계하되, 위 작업을 완료한 프리셋만 오픈한다.

**MVP 프리셋 선정 원칙:** 카테고리 대표성보다 프리셋 완성도와 차별화가 우선이다. 특정 카테고리의 프리셋 품질이 기준에 미달하면 해당 카테고리는 MVP에서 제외하고, 완성도 높은 카테고리에 프리셋을 추가하는 것이 낫다.

### 4-3. 온보딩: 점진적 공개 (Progressive Disclosure)

프리셋 수 자체를 줄이는 것이 아니라, **사용자가 보는 복잡도를 단계적으로 조절**한다.

**1단계 (첫 진입):**
- "어떤 스타일의 투자를 하고 계신가요?" → 카테고리 4~5개만 노출
- 예: 안정적으로 / 성장 중심으로 / 배당 중심으로 / 균형 있게 / 잘 모르겠음
- 선택하면 해당 카테고리의 디폴트 프리셋 자동 적용

**2단계 (코칭 보기 진입 후):**
- 프리셋 드롭다운에서 현재 카테고리의 세부 프리셋 노출
- "성장주를 합리적 가격에 — GARP" 같은 설명형 레이블

**3단계 (설정 또는 탐색):**
- 전체 카테고리 + 전체 프리셋 목록 공개
- 각 프리셋의 철학, 사용 지표, 사용자 설명문 제공

| 구분 | 해결 방식 |
|---|---|
| UX로 해결 | 노출 순서/개수 조절, 설명형 레이블, 단계적 탐색 |
| 구조적으로 결정 | 카테고리당 프리셋 수, 프리셋 간 지표 중복도, 진단 차별화 검증 |

### 4-4. 카테고리 목록

| # | 카테고리 | 코칭 질문 |
|---|---|---|
| 1 | 가치 투자 | "이 가격이 이 기업의 진짜 가치 대비 합리적인가?" |
| 2 | 성장 투자 | "이 기업의 성장이 지속 가능하고, 그 대가로 지불하는 가격이 적정한가?" |
| 3 | 배당/인컴 | "이 기업이 앞으로도 배당을 올릴 수 있는 체력이 있는가?" |
| 4 | 모멘텀/추세 | "시장의 흐름이 이 종목에 우호적인가?" |
| 5 | 퀄리티/방어 | "시장이 흔들릴 때 이 포트폴리오가 버틸 수 있는가?" |
| 6 | 퀀트/팩터 | "어떤 수익 요인(팩터)에 노출되어 있는가?" |
| 7 | 역발상 | "시장이 과잉 반응한 곳에서 기회를 잡고 있는가?" |
| 8 | 테마/전략 | "특정 메가트렌드에 얼마나 효과적으로 노출되어 있는가?" |

## 5. 지표 우선순위 계층

### 5-1. 왜 계층이 필요한가

가중치 매트릭스는 폐기했지만, 모든 지표가 동일한 중요도를 갖는 것은 아니다. GARP에서 PEG는 핵심 철학 그 자체지만, Beta는 참고 수준이다. 이 차이를 반영하지 않으면:

- LLM 코멘트가 핵심과 참고를 동일한 비중으로 언급
- 진단 카드가 사소한 지표의 약점을 주요 문제로 부각
- 사용자가 "이 프리셋에서 정말 중요한 게 뭔데?"를 파악하기 어려움

### 5-2. 3계층 구조 (MVP)

| 계층 | 역할 | 프리셋당 개수 | 예시 (GARP) |
|---|---|---|---|
| **Core** | 이 프리셋의 철학을 정의하는 지표. 여기가 나쁘면 진단 카드의 주요 대상 | 3~5개 | PEG, EPS 성장률, P/E |
| **Supporting** | 핵심 지표를 뒷받침하거나 보완. 코멘트에 포함되지만 카드 대상은 Core 이후 | 3~5개 | ROIC, 부채비율, 매출 성장 지속성 |
| **Context** | 맥락 제공. 이 프리셋의 핵심 판단과는 거리 있음. 펼치기로 확인 | 2~5개 | Beta, 배당수익률, 시가총액 |

### 5-3. Phase 2 확장: Core 세분화

Phase 2에서 Core를 Definitional과 Evaluative로 세분화한다.

| 구분 | 의미 | 미달 시 처리 |
|---|---|---|
| **Definitional Core** | 이 지표가 기준 미달이면 프리셋 대상 자체가 아님 | "이 종목은 이 프리셋과 근본적으로 맞지 않습니다" + 해당 종목 점수 산출 제외 |
| **Evaluative Core** | 매우 중요하지만 기준 미달이 프리셋 부적합을 의미하지는 않음 | "개선이 필요합니다" 코칭 |

예시:
- Dividend Growth에서 "배당 지급 여부" = Definitional (무배당 종목은 대상 아님)
- Dividend Growth에서 "배당성장률" = Evaluative (낮을 수 있지만 대상은 맞음)
- GARP에서 "EPS 성장률 양수 여부" = Definitional (역성장이면 GARP의 G가 성립 안 함)
- GARP에서 "PEG 수준" = Evaluative (높을 수 있지만 대상은 맞음)

**MVP에서는:** Core 기준 미달 시, 내부 산식상 극저점(레벨 1)으로 처리하되, **UI 해석은 분리**한다. 사용자에게는 단순한 저점이 아니라 "이 프리셋과의 적합성이 낮습니다" 또는 "해석 주의" 태그를 별도로 표시한다. "아주 나쁘다"와 "이 렌즈의 적용 대상이 아니다"는 다른 의미이며, 이 구분이 없으면 사용자가 오해할 수 있다. Phase 2에서 Definitional Core 도입 시 "해당 없음" 정식 로직으로 전환.

### 5-4. 계층이 반영되는 곳

**진단 카드 생성:**
- Core 지표 중 약점이 있으면 최우선으로 카드 생성
- Core에 문제 없으면 Supporting으로 내려감
- Context 지표는 단독으로 카드를 생성하지 않음

**LLM 코멘트 톤:**
- Core: 구체적, 프리셋 철학과 연결, 행동 제안 포함
- Supporting: 현황 진술 + 모니터링 권고
- Context: 별도 코멘트 없이 수치만 표시

**UI 노출:**
- Core + Supporting: 기본 화면에 항상 표시
- Context: "더 보기" 뒤에 숨김

### 5-5. 프리셋 차별화가 계층으로 드러남

같은 지표라도 프리셋에 따라 계층이 달라진다.

| 지표 | GARP | Buffett Quality Value | Dividend Growth |
|---|---|---|---|
| PEG | **Core** | Supporting | Context |
| ROE | Supporting | **Core** | Supporting |
| 배당성장률 | Context | Context | **Core** |
| P/E | **Core** | Supporting | Context |
| ROIC 지속성 | Supporting | **Core** | Context |
| 배당성향 | Context | Context | **Core** |

사용자가 프리셋을 바꾸면 같은 종목이라도 강조되는 지표가 바뀌고, 진단 카드의 내용이 달라진다.

## 6. Coach 분석 계층

### 6-1. 왜 계층화하는가

Portfolio Coach는 Chain Sight, Thesis Control, Dashboard 매크로 데이터와 연결되면 강력해진다. 그러나 초기부터 이 모든 모듈에 의존하면 독립 제품으로서의 첫 가치 제공이 불가능하다.

### 6-2. Layer 1: 독립 분석 (MVP)

Chain Sight, Thesis Control, 매크로 데이터 없이 FMP 재무/가격 데이터만으로 작동.

| 제공 기능 | 데이터 소스 |
|---|---|
| 프리셋별 지표 진단 | FMP 재무제표 + 가격 |
| 산업 대비 퍼센타일 | FMP 배치 데이터 |
| 지표별 LLM 코멘트 | 퍼센타일 + 프리셋 계층 |
| 진단 카드 3개 | Core 지표 약점 기반 |
| 강점/약점 정렬 | 레벨 태그 합산 |
| 상태 배지 | 5단계 정합도 |

**MVP에서 사용자가 얻는 가치:**
- "내 포트폴리오가 GARP 관점에서 어떤 상태인지" 즉시 확인
- 업종 대비 강한 점/약한 점 파악
- 프리셋을 바꿔가며 다른 관점으로 같은 포트폴리오 진단
- 분석 저장 후 시간 경과에 따른 비교

### 6-3. Layer 2: 연결 강화 분석 (Phase 2)

다른 모듈과 연결하여 Layer 1에서 볼 수 없는 인사이트 추가.

| 연결 모듈 | 추가 분석 |
|---|---|
| Chain Sight | 관계 클러스터 기반 실질 분산도, 공급망 중복 리스크, HRP 적용 |
| Thesis Control | thesis 건강도, 전제 유효성, Black-Litterman 확신도 기반 비중 적정성 |
| Dashboard 매크로 | 매크로 환경 적합도, 금리 민감도, 경기방어/민감 비율 |
| History | BPT 행동 편향 분석 (처분 효과, 조기 익절 패턴 등) |

### 6-4. Layer 간 UX 처리

Layer 2 데이터가 없을 때 에러가 아닌 **분석 확장 제안**으로 처리한다. 단순 안내가 아니라, **현재 분석의 빈칸과 직접 연결**하여 확장 분석의 가치가 자연스럽게 드러나도록 한다.

| | 표현 | 톤 |
|---|---|---|
| ✅ 적절 | "현재 포트폴리오는 종목 수는 분산되어 보이지만, 관계 기반 집중도는 아직 확인되지 않았습니다. Chain Sight를 연결하면 공급망/경쟁 집중 리스크를 확인할 수 있습니다." | 현재 빈칸 → 확장 가치 연결 |
| ✅ 적절 | "thesis를 설정하면 투자 근거와 포트폴리오 구조의 일치 여부를 점검할 수 있습니다." | 기능 연결 안내 |
| ❌ 부적절 | "프리미엄 기능을 잠금 해제하세요" | 판촉 톤 |
| ❌ 부적절 | "이 기능은 업그레이드가 필요합니다" | 잠긴 기능 느낌 |

"설명 가능한 코치"라는 제품 정체성에 맞게, Layer 2 안내도 "현재 분석에서 아직 보지 못한 부분이 있으며, 이를 확인할 수 있는 방법이 있습니다" 톤을 유지한다.

## 7. 진단 출력 구조

### 7-1. 메인 출력

종합 점수를 메인 KPI로 사용하지 않는다. 숫자 하나에 사용자가 과도하게 집착하면 코칭의 본질이 흐려진다.

```
┌──────────────────────────────────────────────┐
│ 한 줄 상태 요약 (LLM 생성)                     │
│ "GARP 관점에서 성장성은 우수하지만,             │
│  밸류에이션이 업종 대비 높은 편입니다."           │
│                                              │
│ 상태 배지: [정합도 양호 ▲]                     │
├──────────────────────────────────────────────┤
│ 강점 3개               │ 약점 3개              │
│ 🟢 ROIC 업종 상위 12%  │ 🔴 PEG 업종 하위 30%  │
│ 🟢 EPS성장 상위 15%    │ 🔴 부채비율 하위 35%   │
│ 🟢 매출성장 상위 20%   │ 🟡 P/E 업종 상위 75%   │
├──────────────────────────────────────────────┤
│ 진단 카드 3개 (Core 약점 기반)                  │
│ [카드 1] [카드 2] [카드 3]                      │
├──────────────────────────────────────────────┤
│ 지표 상세 (레벨 합산 순 정렬)                    │
│ Core → Supporting → (더 보기: Context)          │
└──────────────────────────────────────────────┘
```

### 7-2. 상태 배지 (5단계)

| 레벨 | 내부 용어 | UI 표현 (사용자 언어) | 의미 |
|---|---|---|---|
| 5 | 정합도 매우 높음 | 매우 잘 맞음 | Core 지표 대부분이 업종 상위권. 프리셋 철학과 포트폴리오가 매우 잘 일치 |
| 4 | 정합도 높음 | 잘 맞음 | Core 지표가 대체로 양호. 일부 개선 여지 |
| 3 | 정합도 보통 | 일부 맞음 | Core 지표가 혼재. 강점과 약점이 공존 |
| 2 | 정합도 낮음 | 잘 맞지 않음 | Core 지표 다수가 업종 하위권. 프리셋 관점에서 구조적 개선 필요 |
| 1 | 정합도 매우 낮음 | 거의 맞지 않음 | Core 지표 대부분이 업종 하위권. 프리셋 변경 또는 포트폴리오 재구성 검토 필요 |

설계 문서에서는 "정합도"를 사용하되, 실제 UI 구현 시에는 사용자 언어("잘 맞음/맞지 않음")로 표현한다.

**산출:** Core 지표의 레벨 태그 평균으로 결정. 상세 산출 방식은 추후 검증 후 확정.

**Saved Analysis 비교 시 활용:**
- "3개월 전: 정합도 보통 → 현재: 정합도 높음" → 포트폴리오 개선 방향이 맞다는 확인
- "프리셋 변경: GARP 정합도 높음 → Buffett 정합도 보통" → 이 포트폴리오는 GARP에 더 가까움

### 7-3. 진단 카드 구조

단순 경고가 아니라, **설명 책임이 있는 카드**로 설계한다.

각 카드는 다음 4개 요소를 반드시 포함:

| 요소 | 역할 | 예시 |
|---|---|---|
| 무엇이 문제인가 | 팩트 진술 | "PEG 평균이 업종 하위 30%에 위치합니다." |
| 어떤 기준에서 문제인가 | 비교 기준 명시 | "반도체 산업 87개사 대비 퍼센타일 기준." |
| 왜 이 프리셋에서 중요한가 | 프리셋 철학 연결 | "GARP은 성장 대비 합리적 가격을 핵심으로 봅니다. PEG가 높다는 것은 성장이 가격에 이미 반영되었을 가능성을 의미합니다." |
| 예외/트레이드오프 | 맥락 보충 | "다만 NVDA의 경우 AI 인프라 독점적 위치를 고려하면 프리미엄이 정당화될 수 있습니다. thesis에서 이 관점을 검증해보시기 바랍니다." |

**문체 원칙:**
- 단정형("비싸다") 보다 조건형/설명형("업종 대비 높은 편이며, 성장 둔화 시 하방 리스크가 커질 수 있습니다")
- Coach는 판단을 내리는 것이 아니라, 판단에 필요한 정보를 제공하는 역할
- 단일 종목 이상치와 구조적 약점을 구분: "INTC 1개 종목이 극단값" vs "5개 종목 중 4개가 일관되게 하위"

### 7-4. 지표 상세

레벨 합산 순으로 정렬. 각 지표마다:

```
[지표명] — [계층: Core]
포트폴리오 대표 퍼센타일: 업종 내 상위 약 28% (비중 반영 요약값)
비교군: 반도체 산업 87개사 (GICS 기준)

종목별 분포:
  레벨5: NVDA (상위 8%), MSFT (상위 12%)
  레벨4: AMZN (상위 25%)
  레벨3: LLY (상위 45%)
  레벨2: INTC (하위 35%)

[LLM 한 줄 코멘트]
"5개 종목 중 3개가 업종 상위 25% 이내. 다만 INTC의 수익성이 하락세입니다."
```

## 8. 순환 루프

### 8-1. MVP 루프

MVP에서는 Screener와 Chain Sight 연결이 없으므로, 루프는 Coach 내에서 닫힌다.

```
[Coach 진단] → 강점/약점 파악
    → "이 분석 저장" (Saved Analysis)
    → 포트폴리오 조정 (종목 추가/제거/비중 변경)
    → Coach 재진단 → 변화 확인
    → (루프 반복)
```

**MVP에서의 행동 유도:**
- 진단 카드 하단에 "관심 종목을 예비 후보로 등록하면 추가 시 변화를 분석해드립니다"
- 비교 탐색 유도: "다른 프리셋에서 이 포트폴리오가 어떻게 해석되는지 비교해볼 수 있습니다." 또는 "현재 포트폴리오는 성장/퀄리티 관점에서 다르게 보일 수 있습니다."
- Coach가 프리셋 방향을 확정해주는 것이 아니라, 사용자가 비교 탐색을 통해 스스로 판단하도록 유도

### 8-2. 확장 루프 (Phase 2+)

Screener, Chain Sight, Thesis Control이 연결되면 완전한 순환 루프가 작동한다.

```
[Coach 진단] → 약점 발견
    → "이 약점을 보완할 종목을 찾아볼까요?" (행동 유도)
    → Screener (프리셋 기준으로 필터링 — 사용자 주도)
    → Chain Sight (후보 종목의 관계 탐색 — 새로운 관계 노드 생성)
    → Thesis Control (투자 가설 설정)
    → Portfolio 예비 후보 등록
    → Coach 재진단 ("이 종목 추가 시 변화 분석")
    → (루프 반복)
```

### 8-3. 락인 메커니즘

확장 루프를 반복할수록:
- Chain Sight의 관계 맵이 사용자별로 고유하게 확장
- Thesis Control의 투자 가설 히스토리가 축적
- Saved Analysis의 진단 히스토리가 쌓여 사후 분석 가치 증가

이 데이터는 사용자가 다른 서비스로 이동하면 처음부터 다시 구축해야 한다.

## 9. 데이터 설계

### 9-1. 퍼센타일 배치 현실화

**초기 유니버스 제한 전략:**

| Phase | 유니버스 | 산출 방식 |
|---|---|---|
| MVP | S&P 500 + 사용자 보유 종목의 산업 peers | Core 지표만 사전 배치 계산 |
| Phase 2 | Russell 3000 | Core + Supporting 사전 배치 |
| Phase 3 | 글로벌 확장 | 전체 사전 배치 + 캐시 |

**온디맨드 계산:**
- S&P 500 외 종목 입력 시, 산업 peers를 실시간 조회하여 퍼센타일 산출
- 캐시 TTL: 24시간 (EOD 배치와 동기화)

### 9-2. 버전 관리

| 버전 항목 | 설명 | 변경 빈도 |
|---|---|---|
| `preset_version` | 프리셋의 지표 구성, 계층, 적정 범위 변경 | 분기 1회 이하 |
| `metric_definition_version` | 지표의 산식, 방향성, 비교군 규칙 변경 | 분기 1회 이하 |
| `scoring_version` | 레벨 태그 기준(퍼센타일 구간) 변경 | 연 1회 이하 |
| `prompt_version` | LLM 코멘트/카드 생성 프롬프트 변경 | 수시 |
| `universe_version` | 퍼센타일 산출 대상 유니버스 변경 | Phase 전환 시 |

Saved Analysis에는 분석 시점의 모든 버전을 함께 저장.

### 9-3. 결측치 정책

| 상태 | 의미 | UI 처리 |
|---|---|---|
| Missing | 데이터 소스에 값 없음 | "데이터 없음" 표시. 해당 지표 정렬에서 제외 |
| Not Applicable | 해당 지표가 이 종목에 적용 안 됨 | "해당 없음" 표시. 별도 설명 |
| Delayed | 데이터 미갱신 (실적 발표 직후 등) | 이전 값 + "갱신 대기" 태그 |
| Insufficient History | 상장 기간 짧아 데이터 부족 | 가용 기간으로 계산 + "제한적 데이터" 태그 |
| Temporarily Unstable | 일시적 이상치 (M&A, 대규모 상각 등) | 값 표시 + "일시적 왜곡 가능" 태그 |

결측 종목의 레벨 합산: 해당 종목 제외, 만점 기준을 (참여 종목 수 × 5)로 조정.

### 9-4. Metric Dictionary 구조 (다음 단계 작성)

| 필드 | 설명 |
|---|---|
| `metric_id` | 고유 식별자 |
| `display_name` | 사용자 표시명 |
| `description` | 설명 (1~2문장) |
| `formula` | 산출 공식 |
| `data_source` | FMP 필드명 또는 계산 로직 |
| `direction` | `higher_is_better` 또는 `lower_is_better` |
| `comparison_group` | 기본 비교군 |
| `fallback_rule` | 비교군 표본 부족 시 fallback |
| `refresh_frequency` | 갱신 주기 |
| `missing_policy` | 결측 시 처리 |
| `display_condition` | 표시 조건 |
| `applicable_presets` | 사용하는 프리셋 목록 |
| `tier_per_preset` | 프리셋별 계층 |

### 9-5. 도메인 엔티티 초안

```
Portfolio (실제 포트폴리오 — 트랙 A)
 ├── Holding (보유 종목)
 │    └── Stock
 └── CandidateHolding (예비 후보)
      └── Stock

Preset (프리셋 정의)
 ├── preset_id, name, category, description, version
 └── PresetMetric (프리셋-지표 매핑)
      ├── metric_id
      ├── tier (Core / Supporting / Context)
      └── direction_override (프리셋별 방향성 재정의, 필요 시)

Metric (지표 정의 — Metric Dictionary)
 ├── metric_id, display_name, formula, direction
 ├── comparison_group, fallback_rule
 └── missing_policy

AnalysisRun (분석 실행)
 ├── run_id, portfolio_id, preset_id, timestamp
 ├── version_bundle (preset_v, metric_v, scoring_v, prompt_v, universe_v)
 ├── portfolio_hash (구성 해시)
 ├── status_badge (5단계)
 └── MetricResult (지표별 결과)
      ├── metric_id, stock_id
      ├── raw_value (원시 값)
      ├── percentile (산업 대비 퍼센타일)
      ├── level_tag (1~5)
      ├── comparison_meta (비교군 정보, 표본 수, fallback 여부)
      └── data_status (정상/Missing/N-A/Delayed/Insufficient/Unstable)

SavedAnalysis (저장된 분석 — 영구)
 ├── saved_id, analysis_run_id, saved_at, user_memo
 └── AnalysisRun 전체 데이터 스냅샷 (불변)

TempAnalysis (임시 저장 — 자동, 중복 제거)
 ├── temp_id, analysis_run_id, created_at
 ├── dedup_key (portfolio_hash + preset_id)
 └── AnalysisRun 전체 데이터 스냅샷
 * 동일 dedup_key 존재 시 최신 결과로 덮어쓰기
 * 포트폴리오 구성 변경 시 새로 생성
 * 최대 보관 수 초과 시 가장 오래된 것부터 삭제

DiagnosticCard (진단 카드)
 ├── card_id, analysis_run_id, priority (1~3)
 ├── target_metric_id
 ├── problem_statement
 ├── comparison_basis
 ├── preset_relevance
 ├── exception_tradeoff
 └── generated_by (prompt_version)

LLMComment (지표별 코멘트)
 ├── comment_id, analysis_run_id, metric_id
 ├── comment_text
 └── generated_by (prompt_version)

PercentileCache (퍼센타일 배치 캐시)
 ├── metric_id, industry_code, date
 ├── distribution_data (백분위 구간별 값)
 └── sample_size
```

**엔티티 간 관계:**
- Portfolio 1:N Holding
- Preset 1:N PresetMetric
- AnalysisRun → Portfolio, Preset
- AnalysisRun 1:N MetricResult
- AnalysisRun 1:N DiagnosticCard (최대 3개)
- AnalysisRun 1:N LLMComment
- SavedAnalysis 1:1 AnalysisRun
- TempAnalysis 1:1 AnalysisRun

| 항목 | 상세 |

| 결측치 처리 | 5가지 상태 분류 |
