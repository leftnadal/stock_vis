# Stock-Vis 작업 규약 — 참조 문서 (업로드용)

> **이 파일을 어디에 쓰나:** 각 앱 Claude Project에 **파일로 업로드**한다(프로젝트 지식). 매 대화에 자동 주입되는 건 커스텀 지시문의 *공용 코어*이고, 이 문서는 디렉터(채팅 Claude)가 _특정 작업에서 깊이가 필요할 때 펼쳐 보는_ 참조다. 휘발성(브랜치·HEAD·경로·수치)은 담지 않는다 — 그건 세션마다 측정한다.

---

## 1. 의존 방향 규칙 — 왜 한 방향인가

`apps → shared`만 허용. 핵심 직관: **재료는 자기가 어느 요리에 들어갈지 몰라야 한다.** shared가 특정 앱을 알기 시작하면 "범용 재료"가 아니라 "그 앱에 묶인 부품"이 되어, 그 앱이 바뀔 때 shared와 shared를 쓰는 모든 앱이 연쇄로 흔들린다. 그래서 shared는 누구도 모르게 유지한다.

- **외부 API(FMP 데이터·LLM 제공자)는 `packages/shared`의 래퍼 경유로만 접근한다**(예: FMP client, LLMClient 래퍼, circuit_breaker). 앱이 외부를 _직접_ 때리지 않는다 — 의존 방향 규칙의 연장(외부 의존도 토대에 모은다). 구체 엔드포인트·rate limit·제공자 정책·통합 상태는 ground truth → STEP 0/코드/DECISIONS에서 확인(여기 박지 않는다).

## 2. 경계 가드 메커니즘

- 아키텍처 테스트(`tests/architecture/test_shared_boundary.py` 계열)가 `packages/shared` 안의 `from apps.* import`를 **AST로 스캔**해 위반을 잡는다.
- `KNOWN_VIOLATIONS` = **동결 목록**. 이미 알고 의도적으로 둔 위반은 통과시키되, _새_ 위반은 막는다("우회 0").
- 동결 갱신은 **테스트와 `scripts/health_check.py` 양쪽을 동시에** 한다(한쪽만 고치면 어긋남).
- "동결 N"은 무해하게 묶인 위반 수. 0으로 줄이는 과정 = **burn-down**.

## 3. 경계 위반 청소 방향 패턴 (재사용)

shared가 어떤 앱을 거꾸로 import할 때, 청소 후보는 셋. 무엇을 고를지는 **STEP 0 실측**으로 정한다(추측 금지).

- **방향1 — 소비자 이동:** shared 안의 그 코드가 사실 특정 앱 전용이면, 그 코드를 그 앱으로 옮긴다 → `app→app`이라 합법. _모델이 아니라 소비자를 옮긴다._ prod DB 무관.
- **방향2 — 의존 역전(주입):** shared는 모델을 import하지 않고, 호출측이 필요한 데이터를 인자로 주입. 그 코드가 여러 앱 공유일 때.
- **방향C — 토대 승격:** 그 모델이 범용 토대 데이터면 shared로 올린다. 단 app_label·테이블 보존 이동인지(무마이그레이션) vs 테이블 변경(prod 대공사)인지 `makemigrations --dry-run`으로 반드시 검증.
- **함정:** 모델을 다른 앱으로 옮기는 건 화살표 타겟만 바꿀 뿐 위반은 그대로 + prod 마이그레이션을 만든다. 청소가 아니다.

## 4. worktree 모델

- 작업은 세션마다 **worktree**에서 시작한다. 목적은 **세션·프로젝트 간 git 충돌 방지**(여러 Claude Project가 같은 repo를 다루는 시나리오의 안전장치).
- 패턴: 메인 머지 지점 worktree + 관리(mgmt) 작업용 worktree를 분리 운용.
- 세션 시작 시 `git worktree list`로 위치·동기 상태 확인.
- 브랜치 명명: worktree 브랜치는 `monorepo/sess-X`로 짓는다(commit hook 화이트리스트 `monorepo/*`를 재사용 — `sess/*`는 거부됨).
- (별개 가드) pre-commit hook이 **브랜치명 화이트리스트**로 commit을 게이팅한다. 단 이 hook은 현재 `.git/hooks`(git 버전관리 밖)라 linked worktree끼리는 공유되지만 **새 클론엔 안 따라온다.** → **hardening(c) 보류 중, 트리거: 새 클론/새 기계 OR hook 로직 변경 시 `scripts/hooks` + `git config core.hooksPath`로 이전.**

## 5. 하네스 상세

- `PROGRESS.md`: 진행 상황 **캐시**(진실의 소스 아님 — 코드와 어긋날 수 있음).
- `DECISIONS.md`: 결정과 그 근거 기록. 미래 세션의 오해 방지를 위해 "왜"를 남긴다.
- `TASKQUEUE`: 잔여·보류 작업. 보류는 사라지지 않게 여기 등록.
- `common-bugs.md`: 반복되는 함정과 소진 순서.
- `scripts/health_check.py`: 정합성 검증(여러 항목). 세션 시작·종료 시 통과 확인.
- 정합성 시스템(자동추출/검문소/세션강제/메모리정렬)으로 PROGRESS stale을 방지.
- **관리(mgmt)/ops 세션 범위:** mgmt는 *기능 도메인이 아니라 운영 계층*을 다룬다 — 하네스(위 파일들)·`scripts/`(health_check·nightly 자동화)·`tests/architecture`(경계 가드)·비용 원장·구조/형상. **앱의 기능 코드는 바꾸지 않는다**(옮기고 정돈만, 행위보존). nightly 자동화의 *오케스트레이션*은 mgmt 소유지만 그것이 돌리는 *도메인 파이프라인*은 해당 앱 소유. (mgmt는 별도 프로젝트가 아니라 이 공용 규약으로 다룬다.)

## 6. 세션 계약서 · STARTUP_CHECKLIST · STEP 0

- 세션 시작 의식: STARTUP_CHECKLIST Step 0에서 **계약 헤더**(세션 종류·범위·baseline)를 채운다.
- **STEP 0 = ground truth 측정.** 작업 전 사실관계(브랜치·HEAD·구조·테스트)를 grep/명령으로 확인한다. 이것이 메모리 추정과 코드의 불일치를 막는 핵심이며, 예상 밖 상황(예: 머지된 줄 알았는데 아닌 브랜치)을 잡아낸다.

## 7. git · 커밋 · 명명 규약

- 3계층: **Slice**(검증 단위) / **Part**(실행 묶음) / **Step**(단일 작업).
- 의미 단위 분리 커밋(한 슬라이스 = 여러 커밋). 슬라이스별 브랜치.
- **IDENTICAL hash**: 행위보존 변경을 해시 동일성으로 입증.
- 회귀 격리 KPI(±30%)로 각 Part 검증.

## 8. 결정 방법론 상세

- **운영 원칙 ①:** 선택지의 *내용 자체*를 먼저 쉽게 설명한다(의미·작동방식·시나리오). 가중합 표 앞에 "이게 무슨 선택인가" 단락 필수.
- **운영 원칙 ②:** 가중합 마진 1 초과면 추가 확인 없이 자동 결정.
- 정량 가중합: weights 합=1.00, 1~5 점수, 마진 명시. 마진 0.40 미만=타이브레이커, 0.86 이상=압도적.
- UI/UX와 연결되는 결정은 목업으로 보여 사용자 가치를 설명. 목업으로 설명 안 되는 결정은 꼭 필요한 것만.

## 9. HALT 패턴 (멈추고 보고)

- 빈 스키마 / 응답 봉투 형식 불일치 / 외부 자동화 방어.
- 미검토 작업 발견(예: `git branch -d`가 미머지 브랜치를 거부) → 강제로 뚫지 말고 HALT.
- 지시서 절대 규칙을 넘어야 하는 상황 → 사용자 수동 승인 필요.

## 10. 핵심 원칙 한 줄

**안정적인 것은 문서에 박고, 휘발성인 것은 세션마다 측정한다. 행위보존이 최우선이고, 같은 규약을 N벌 복제하지 않는다(단일 출처 — 복제는 drift를 부른다).**
